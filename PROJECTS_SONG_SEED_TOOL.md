# PROJECTS_SONG_SEED_TOOL.md
## A Better Finish-Line for the 14

Not every project wants the same kind of ending.

Some are **Songs**.  
Some are **Seeds**.  
Some are **Tools**.

The pain begins when a Seed is judged like a Tool, or a Song is forced to justify itself like a product.

---

## Definitions

### Song
A project whose value exists in expression, insight, rhythm, witness, or awakening.

**Finish line:** truthfully expressed.

Examples:
- poem
- performance
- narrative
- coda
- chapter
- sound work
- symbolic map

### Seed
A project whose value lies in preserving a pattern, blueprint, architecture, or future possibility.

**Finish line:** clearly preserved.

Examples:
- concept note
- framework
- prototype
- architecture map
- future-facing spec
- research direction

### Tool
A project whose value lies in solving a concrete task for a real user or operator.

**Finish line:** actually usable.

Examples:
- MVP
- sensor node
- browser extension
- workflow
- operational system
- calculator
- tracker

---

## Working Rule

Do not ask every project to become public, profitable, or fully adopted right now.

Some works only need to be:
- expressed
- preserved
- or made narrowly usable

That is enough.

## Better Finish-Lines

### For Songs
Do not ask: “Will the crowd use this?”
Ask: “Is it honestly expressed?”

### For Seeds
Do not ask: “Will people adopt this now?”
Ask: “Is the blueprint preserved clearly enough for future worthy hands?”

### For Tools
Do not ask: “Will everyone value this?”
Ask: “Does it solve one real job cleanly?”

---

## The Practical Cure for the Final 5%

When a project stalls near the end, ask first:

**Is this a Song, a Seed, or a Tool?**

Then finish it by the right standard:

- **Song** → express it
- **Seed** → seal it
- **Tool** → narrow it until it works

---

## Final Reminder

A work does not fail just because the wrong audience cannot receive it.

A Song may remain true.  
A Seed may remain preserved.  
A Tool may serve only a few.

That is still enough.
